So, you want to get into modding Ostranauts? Welcome aboard! This guide will help you get started with modding, from installing mods to creating your own.

Ostranauts supports two main types of modding:

Data/JSON Modding - Modify game data files to add or change items, interactions, ships, conditions, and more without writing code
C# Code Modding - Use BepInEx and Harmony to patch game code and add entirely new features, behaviors, and systems
Whether you're just looking to tweak some items or build complex new systems, this guide has you covered.

Installing Mods
Before you can start creating mods, you'll want to know how to install them! Ostranauts uses the Thunderstore mod repository and supports several methods of mod installation.

Using r2modman or Gale Mod Manager
The easiest way to install mods is using a mod manager. Both r2modman and Gale work great with Ostranauts:

r2modman:

Download r2modman from Thunderstore
Launch r2modman and select "Ostranauts" from the game list
Browse available mods or search for specific ones
Click "Install" on any mod you want to add
Launch the game through r2modman using the "Start modded" button
Gale:

Download Gale from Thunderstore or its official website
Select Ostranauts as your game
Browse and install mods with a single click
Launch the game through Gale
Tip: Mod managers automatically handle dependencies and load order, making your life much easier!

Manual Installation
For manual installation, you'll need BepInEx installed first:

Download BepInEx 5.x (64-bit) from GitHub
Extract the BepInEx folder into your Ostranauts game directory (where Ostranauts.exe is located)
Run the game once to generate BepInEx folders
Download mods from Thunderstore (usually as .zip files)
Extract mod folders into BepInEx/plugins/
Launch the game
Your folder structure should look like:

Ostranauts/
├── Ostranauts.exe
├── BepInEx/
│   ├── core/
│   ├── plugins/
│   │   ├── YourMod1/
│   │   └── YourMod2/
│   └── config/
└── Ostranauts_Data/
Copy

Common Modding Terms
Before diving into modding, familiarize yourself with these essential terms:

Term	Definition
BepInEx	Mod loader framework that allows mods to run. Required for all Ostranauts mods.
Harmony	Library that patches game code at runtime, enabling code mods to modify game behavior.
JSON	Text file format used for all game data (items, interactions, ships, etc.).
CondOwner	Any game object that can have conditions applied to it (items, crew, ships, rooms).
COOverlay	Represents a specific state or variant of a CondOwner (damaged, loose, different colors, etc.) with its own name, description, and visual appearance. Multiple COOverlays can reference the same CondOwner.
Interaction	An action a character can perform (use, repair, install, hug, etc.).
StreamingAssets	Folder containing all game data files, located at Ostranauts_Data/StreamingAssets/.
Thunderstore	Primary mod repository and distribution platform for Ostranauts.
Socket/Slot	Grid-based placement system that determines where items can be placed.
Condition	Status effect or game state applied to CondOwners (hungry, tired, comforted, etc.).
For a complete glossary of modding terms, see Modding Glossary.

Data Modding
What is Data Modding?
Data modding (also called JSON modding) lets you modify game content by editing or adding JSON files. This is the most accessible form of modding since it requires no programming knowledge—just some patience and attention to detail.

What Can You Do with Data Modding?
Data mods can add or modify:

Items - New objects, tools, containers, furniture, and equipment
Interactions - Actions characters can perform (hug, repair, use, etc.)
Ships - Complete ship designs and templates
Conditions - Status effects and game states (see Modding/CondOwners for details)
Rooms - Room templates and specifications
Loot tables - What items spawn where (see Modding/Loot for details)
Careers - Character background options
AI behaviors - Crew task priorities and decision-making
And much more - See Modding/Data_Modding for a comprehensive list
What Can't Data Modding Do?
Data modding has some limitations:

Cannot add new game mechanics or systems
Cannot modify UI layouts or create new UI elements
Cannot change core game logic or behavior
Cannot add new code-based features
For these advanced changes, you'll need C# code modding.

C# Code Modding
What is C# Code Modding?
C# code modding uses BepInEx (a mod loader) and Harmony (a patching library) to modify the game's compiled code at runtime. This is the most powerful form of modding, allowing you to add entirely new features, change game systems, and create complex behaviors.

What Can You Do with Code Modding?
Code mods enable:

New game mechanics - Completely new systems and features
UI modifications - Custom interfaces, tooltips, and displays
Advanced behaviors - Complex AI, automation, and logic
Quality of life features - Shortcuts, information displays, and conveniences
Integration systems - Connect different game systems in new ways
Data extension - Add fields and capabilities beyond what JSON allows
Prerequisites
Code modding requires some technical knowledge:

C# programming - Basic to intermediate understanding of C#
Visual Studio - Development environment for writing and building mods
Understanding of modding concepts - Harmony patches, BepInEx plugin structure
Patience and debugging skills - Things won't always work the first time!
New to C#? There are many free tutorials online. The language is similar to Java and C++, so if you know those, you're already ahead!

Understanding Ostranauts Data Structure
What is StreamingAssets?
Ostranauts stores all its base game data in a special folder called StreamingAssets, located at: <Game Directory>/Ostranauts_Data/StreamingAssets/

This folder contains:

data/ - All JSON game data files
images/ - All sprite graphics (PNG files)
audio/ - Music and sound effects
mesh/ - 3D models (OBJ format)
Important: Do NOT modify files in StreamingAssets directly! Create mods instead.

See Sprites & Assets Guide for details on working with images and assets.

The Data Folder Structure
Inside StreamingAssets/data/, you'll find over 30 different data types organized by folder:

Folder	Contains	See Also
items/	Item definitions (tools, objects, equipment)	Items
cooverlays/	Display names and descriptions for items	[[Modding/CondOwners]]
interactions/	Actions characters can perform	Interactions
conditions/	Status effects and game states	CondOwners
condowners/	Objects that can have conditions	CondOwners
condrules/	Rules that trigger condition changes	CondOwners
condtrigs/	Condition trigger definitions	CondOwners
loot/	Loot tables for item spawning	Loot System
ships/	Ship templates and designs	Ships
rooms/	Room specifications	-
slots/	Socket/slot system definitions	-
careers/	Character background options	-
pledges/	AI behavior definitions	Pledges
jobs/	Task definitions	-
installables/	Objects that can be installed	-
Tip: Browse the base game's data files to see examples of how everything works!

Finding Your Mods Folder
Locating the Mods Folder
The Mods folder is located in your Ostranauts installation directory:

Default location: <Steam>/steamapps/common/Ostranauts/Ostranauts_Data/Mods/

To find it in-game:

Launch Ostranauts
Go to Options → Files
Click the Mods button to open the folder
The game will create the folder structure if it doesn't exist
Note: This is NOT the same as BepInEx/plugins/. BepInEx plugins are for C# code mods, while the Mods folder is for JSON data mods.

Creating Your First Mod
Mod Folder Structure
Your mod should have this structure:

Ostranauts_Data/
├── StreamingAssets/
└── Mods/
    ├── loading_order.json
    ├── YourModName/
    │   ├── mod_info.json
    │   ├── data/
    │   │   ├── items/
    │   │   │   └── my_items.json
    │   │   ├── cooverlays/
    │   │   │   └── my_overlays.json
    │   │   ├── interactions/
    │   │   │   └── my_interactions.json
    │   │   └── condtrigs/
    │   │       └── my_condtrigs.json
    │   └── images/
    │       ├── ItmMyItem.png
    │       └── ItmMyItemn.png
Copy
Required Files
mod_info.json
Each mod folder must contain a mod_info.json file with metadata about your mod:

[
  {
    "strName": "My Cool Mod",
    "strAuthor": "YourName",
    "strModURL": "https://github.com/yourname/yourmod",
    "strGameVersion": "1.3.1",
    "strModVersion": "1.0.0",
    "strNotes": "A short description of what your mod does."
  }
]
Copy
Field descriptions:

strName - Display name of your mod
strAuthor - Your name or username
strModURL - Link to your mod's website/repository
strGameVersion - Game version this mod is for
strModVersion - Your mod's version number
strNotes - Brief description
loading_order.json
The loading_order.json file goes in Ostranauts_Data/ (one level up from the Mods folder) and controls which mods load and in what order:

[
  {
    "strName": "Mod Loading Order",
    "strNotes": "Controls the order mods are loaded. 'core' refers to base game data and should usually be first.",
    "aLoadOrder": [
      "core",
      "YourModName",
      "AnotherMod"
    ],
    "aIgnorePatterns": []
  }
]
Copy
Important notes:

"core" represents the base game and should typically be first
Mods load in the order listed
Later mods override earlier ones if they define the same strName
Mod folder names in aLoadOrder must match exactly (case-sensitive!)
If you click the Mods button in Options→Files, the game will create this file for you
JSON File Structure
JSON Basics
JSON (JavaScript Object Notation) is a simple data format using key-value pairs. Here's a quick primer:

[
  {
    "strName": "MyItem",
    "strDescription": "A description",
    "nValue": 100,
    "bEnabled": true,
    "aList": ["item1", "item2", "item3"]
  }
]
Copy
Key rules:

Files contain arrays ([ ]) of objects ({ })
Strings use double quotes: "text"
Numbers don't need quotes: 100
Booleans are true or false (no quotes)
Arrays contain comma-separated values: ["a", "b", "c"]
Every item needs a comma except the last one
Field names (keys) are case-sensitive!
Ostranauts Naming Conventions
Ostranauts uses prefixes to indicate data types:

str = String (text)
n = Number (integer)
f = Float (decimal number)
b = Boolean (true/false)
a = Array (list)
map = Map/dictionary (key-value pairs)
Examples:

strName - The name (string)
nCols - Number of columns (integer)
fZScale - Z-axis scale (float)
bLerp - Whether to lerp/interpolate (boolean)
aSocketAdds - Array of socket additions
How Mod Loading Works
Loading Order
Ostranauts loads data in this order:

Base game data from StreamingAssets/data/ ("core")
Mod data in the order specified in loading_order.json
Each mod's data/ folder follows the same structure as StreamingAssets/data/.

Overriding vs. Adding
Adding new content: Use unique strName values that don't exist in the base game.

[
  {
    "strName": "ItmMyNewItem",
    "strImg": "ItmMyNewItem",
    ...
  }
]
Copy
Overriding existing content: Use the same strName as the base game item you want to replace.

[
  {
    "strName": "ItmToyDinosaur01",
    "strImg": "ItmMyCustomSprite",
    ...
  }
]
Copy
Warning: Overriding base game content can cause conflicts with other mods!

Common Data Types to Mod
Items (items/)
Items are objects in the game world. See the sample mod tutorial for a complete example.

Key fields:

strName - Unique identifier
strImg - Base sprite name
strImgNorm - Normal map sprite name
strImgDamaged - Damaged state sprite
nCols - Width in grid columns
fZScale - Visual height/depth
aSocketAdds - What sockets this item adds
aSocketForbids - What sockets prevent placement
aSocketReqs - What sockets are required
Example:

[
  {
    "strName": "ItmMyTool",
    "strImg": "ItmMyTool",
    "strImgNorm": "ItmMyTooln",
    "strImgDamaged": "",
    "strDmgColor": "DamageTintDefault",
    "fZScale": 0.02,
    "nCols": 1,
    "aSocketAdds": ["TILItemAdds"],
    "aSocketForbids": [
      "Blank", "Blank", "Blank",
      "Blank", "TILItemForbids", "Blank",
      "Blank", "Blank", "Blank"
    ],
    "aSocketReqs": [
      "Blank", "Blank", "Blank",
      "Blank", "Blank", "Blank",
      "Blank", "Blank", "Blank"
    ]
  }
]
Copy
COOverlays (cooverlays/)
COOverlays provide the human-readable names and descriptions that appear in tooltips. Every item needs a matching COOverlay!

Key fields:

strName - Must match item's strName
strNameFriendly - Display name
strDesc - Description text
strImg - Display sprite (usually same as item)
strPortraitImg - Icon for inventory
strCOBase - Base CondOwner reference
strCondLoot - Loot table containing conditions to apply to this overlay
mapModeSwitches - Defines how states transition between variants
aInteractionsReplace - Array of interaction replacements (pairs: old, new)
Example:

[
  {
    "strName": "ItmMyTool",
    "strNameFriendly": "My Awesome Tool",
    "strDesc": "A tool for doing awesome things.",
    "strImg": "ItmMyTool",
    "strImgNorm": "ItmMyTooln",
    "strPortraitImg": "ItmMyTool",
    "strCOBase": "ItmTool01"
  }
]
Copy
Interactions (interactions/)
Interactions define actions characters can perform. See Interactions for detailed documentation.

Key fields:

strName - Unique identifier
strTitle - Display title
strDesc - Description with pronoun tokens
strAnim - Animation to play
fDuration - How long it takes (in game hours)
LootCTsUs - Condition triggers applied to the actor (character performing action)
LootCTsThem - Condition triggers applied to the target object
Pronoun tokens in descriptions:

[us] - The character performing the action
[them] - The target of the action
[us-subj] - Subject form (he/she/they)
[us-obj] - Object form (him/her/them)
[us-pos] - Possessive (his/her/their)
Example:

[
  {
    "strName": "ACTHugToy",
    "strTitle": "Hug",
    "strDesc": "[us] [hugs] [them] tightly.",
    "strActionGroup": "Social",
    "strAnim": "Idle",
    "strIdleAnim": "Idle",
    "fDuration": 0.001,
    "bIgnoreFeelings": true,
    "strThemType": "Other",
    "LootCTsUs": "CTHugComfort",
    "LootCTsThem": null
  }
]
Copy
Conditions (conditions/)
Conditions are status effects applied to CondOwners. They can represent physical states, emotional states, or game flags.

Key fields:

strName - Unique identifier
strNameFriendly - Display name
strDesc - Description
strColor - Color category (Good/Bad/Neutral)
fDuration - How long it lasts (0 = permanent)
nDisplaySelf - Show to owner? (0=no, 1=yes, 2=verbose)
nDisplayOther - Show to others?
bFatal - Does this kill the owner?
Example:

[
  {
    "strName": "Comforted",
    "strNameFriendly": "Comforted",
    "strColor": "Good",
    "strDesc": "[us] [feels] comforted and safe.",
    "fDuration": 0.5,
    "nDisplaySelf": 2,
    "nDisplayOther": 1,
    "bFatal": false
  }
]
Copy
Condition Triggers (condtrigs/)
Condition triggers (CondTrigs) are events that fire based on condition states. They're used extensively in interactions via LootCTsUs and LootCTsThem fields.

Key fields:

strName - Unique identifier
strCondName - Condition name to apply
aReqs - Array of required conditions to test
aForbids - Array of forbidden conditions to test
aTriggers - Array of condition triggers to fire
aTriggersForbid - Array of condition triggers that prevent firing
fChance - Probability of triggering (0.0 to 1.0)
fCount - Amount of condition to apply
bConsumeSource - Whether to consume the source item
Example:

[
  {
    "strName": "CTAddComfort",
    "aCondsToTest": [],
    "aCondsToAdd": [
      "Comforted,0.5"
    ],
    "aCondsToRemove": [],
    "fResolveChance": 1.0,
    "bTestAny": false,
    "bConsumeSource": false
  }
]
Copy
See Stuffed Animal Tutorial for a practical example.

Understanding Harmony
Harmony is a library that lets you patch (modify) game code at runtime without changing the original game files. Instead of directly editing Assembly-CSharp.dll, you inject your code into methods as they execute.

Harmony patch types:

Prefix - Runs before the original method
Postfix - Runs after the original method
Transpiler - Modifies the method's IL code (advanced)
Finalizer - Runs after the method, even if it throws exceptions
For complete documentation, see Harmony Documentation.

Exploring Game Code with dnSpy
What is dnSpy?
dnSpy is a .NET debugger and assembly editor that lets you decompile and explore the game's code. This is essential for modding because you need to:

Find methods to patch
Understand method signatures (parameters, return types)
See how game systems work
Identify the right classes and fields to modify
Installing dnSpy
Download from GitHub Releases
Get the latest dnSpy-net-win64.zip (for 64-bit Windows)
Extract to a folder (e.g., C:\Tools\dnSpy\)
Run dnSpy.exe (no installation needed!)
Note: dnSpy is no longer actively maintained, but it still works perfectly for Unity games like Ostranauts.

Understanding Method Signatures
When you find a method to patch, note its signature carefully:

```csharp public static Item SpawnItem(string strItemDef, Vector3 v3Pos, Quaternion qRot, Transform tParent) ```

Key information:

Access modifier: public static
Return type: Item
Method name: SpawnItem
Parameters: string, Vector3, Quaternion, Transform
You need this exact signature when creating Harmony patches!

Reading Decompiled Code
dnSpy shows you C# code reconstructed from IL bytecode:

Tips for reading:

Variable names may be generic (text, num, flag)
Some logic may look awkward (it's reconstructed)
Focus on what it does, not how it's written
Look for field names, method calls, and data flow
What to look for when patching:

Method entry points (where to add Prefix)
Method exit points (where to add Postfix)
Fields you need to access (__instance.fieldName)
Other methods being called (potential patch targets)
Finding Fields and Properties
Click on a class to see its members in the right panel:

Fields: Data stored in the class

Example: Item.strName - The item's internal name
Example: Character.fHealth - Character health value
Properties: Computed or wrapped values

Example: CondOwner.FriendlyName - Display name
May have getters/setters you can hook into
Methods: Functions you can patch

Look for void Awake(), void Update(), etc.
Game lifecycle methods are common patch targets
Analyzing Harmony Patch Targets
Good methods to patch:

Awake() - When object initializes
Start() - When object starts (after Awake)
Update() - Called every frame
Data processing methods (LoadData, SaveData)
Action methods (UseItem, TakeDamage)
Avoid patching:

Property getters called frequently (performance impact)
Internal Unity methods
Very short methods (may be inlined)
Generic methods without specific types

Understanding Special Harmony Parameters
When viewing methods in dnSpy, you can access them in patches:

__instance - The object instance

// In dnSpy: public string strName;
// In patch:
Item __instance;
Logger.LogInfo(__instance.strName);  // Access the field
Copy
__result - The return value (Postfix only)

// In dnSpy: public Item SpawnItem(...)
// In patch:
[HarmonyPostfix]
static void Postfix(Item __result)
{
    Logger.LogInfo($"Spawned: {__result.strName}");
}
Copy
__args - Array of all parameters

// In dnSpy: public void Method(string a, int b, bool c)
// In patch:
static void Prefix(object[] __args)
{
    string a = (string)__args[0];
    int b = (int)__args[1];
    bool c = (bool)__args[2];
}
Copy

Copy
Change GameDir to your Ostranauts installation path!

3. Edit the main plugin file:

Open YourModName.cs and update the BepInPlugin attribute:

[BepInPlugin("com.yourname.yourmod", "Your Mod Name", "1.0.0")]
public class YourModPlugin : BaseUnityPlugin
{
    internal static new ManualLogSource Logger;
    private Harmony _harmony;

    private void Awake()
    {
        Logger = base.Logger;
        Logger.LogInfo($"Plugin com.yourname.yourmod is loaded!");

        // Apply Harmony patches
        _harmony = new Harmony("com.yourname.yourmod");
        _harmony.PatchAll(typeof(YourModPlugin).Assembly);
        
        Logger.LogInfo("Harmony patches applied successfully!");
    }

    private void OnDestroy()
    {
        _harmony?.UnpatchSelf();
    }
}
Copy
Key fields:

First parameter: Unique mod ID (use reverse domain notation)
Second parameter: Display name
Third parameter: Version number

Creating Your First Harmony Patch
Let's create a simple patch that logs when items are spawned. This demonstrates the basics without breaking anything.

Create a Patches Folder
Right-click your project in Solution Explorer
Add → New Folder → Name it "Patches"
Right-click Patches folder
Add → Class → Name it ItemPatches.cs
Write the Patch
using HarmonyLib;
using BepInEx.Logging;

namespace YourNamespace.YourModName.Patches
{
    [HarmonyPatch(typeof(Item))]
    public static class ItemPatches
    {
        [HarmonyPatch("Awake")]
        [HarmonyPostfix]
        static void AwakePostfix(Item __instance)
        {
            YourModPlugin.Logger.LogInfo($"Item spawned: {__instance.strName}");
        }
    }
}
Copy
Understanding the Patch
[HarmonyPatch(typeof(Item))] - Targets the Item class
[HarmonyPatch("Awake")] - Targets the Awake method
[HarmonyPostfix] - Runs after the original method
__instance - Special parameter containing the Item instance
See Glossary for more on patches and CondOwners for game object types.

Understanding OstraAutoloader
OstraAutoloader is a mod that simplifies mod loading in Ostranauts. Instead of manually managing loading_order.json, it:

Automatically discovers mods in both Ostranauts_Data/Mods/ and BepInEx/plugins/
Resolves mod dependencies automatically
Generates loading_order.json for you
Enables mod managers (Gale, r2modman) to work properly
Why use it?

For users: Mod managers can install mods automatically
For mod authors: No manual JSON editing required
For everyone: Automatic dependency handling prevents load order issues
Learn more about OstraAutoloader

Creating Autoload.Meta.toml
If your mod includes JSON data files or has dependencies, create an Autoload.Meta.toml file in your mod's root folder:

[AutoLoad]
Manifest = "manifest.json"
LoadGroup = "Default"

[AutoLoad.HardDependencies]
# These mods MUST be present and load first
# Example: "OtherModName" = "1.0.0"

[AutoLoad.SoftDependencies]  
# These mods load first IF present (optional)
# Example: "OptionalMod" = "1.0.0"
Copy
Load Groups:

Default - Standard mods
BeforeFFU - Load before FFU (rarely needed)
AfterFFU - Load after FFU (if your mod modifies FFU content)
Note: If you don't have JSON data or dependencies, you may not need this file!

Creating thunderstore.toml
This file configures your mod for Thunderstore:

[package]
namespace = "YourTeamName"
name = "YourModName"
versionNumber = "1.0.0"
description = "A short, compelling description of your mod"
websiteUrl = "https://github.com/yourusername/yourmod"

[package.dependencies]
BepInEx-BepInExPack = "5.4.21"
Robyn-OstraAutoloader = "0.1.10"  # Recommended for compatibility

[build]
icon = "icon.png"
readme = "README.md"
outdir = "build"

[[build.copy]]
source = "bin/Debug/netstandard2.1/YourModName.dll"
target = "plugins/YourModName.dll"

# If you have JSON data mods
[[build.copy]]
source = "data"
target = "plugins/YourModName/data"

# If you have Autoload config
[[build.copy]]
source = "Autoload.Meta.toml"
target = "plugins/YourModName/Autoload.Meta.toml"
Copy
Key fields:

namespace - Your team name on Thunderstore
name - Mod name (alphanumeric and underscores only)
versionNumber - Must be semantic versioning (X.Y.Z)
dependencies - Other mods required (always include BepInEx!)
source - Path to your compiled DLL
target - Where it goes in the mod package (use plugins/ for Autoloader compatibility)
Important: Note that target uses plugins/YourModName.dll not BepInEx/plugins/. When packaging for Thunderstore, the structure is different from local installation!

Automated CI/CD with GitHub Actions
What is CI/CD?
Continuous Integration/Continuous Deployment automates building and publishing your mod. When you create a release on GitHub, it:

Downloads encrypted game reference assemblies
Builds your mod DLL automatically
Packages it for Thunderstore
Publishes to Thunderstore automatically
All without touching your local machine!
Benefits:

No manual build/upload steps
Consistent, reproducible builds
Build from any computer
Automatic version management
Understanding the Workflow
The .github/workflows/release.yml file defines the automation. See the template's workflow file for the complete example.

Key workflow features:

Triggers manually or on release creation
Downloads and decrypts game assemblies
Builds your mod DLL
Packages for Thunderstore (including Autoload files)
Publishes automatically
Workflow inputs:

Version number (optional - can detect from git tags)
Reference assemblies filename (which game version to build against)

How the Workflow Works
When you trigger the release workflow:

Step 1: Setup

Checks out your code
Determines version from input or git tags
Sets up .NET SDK
Step 2: Download Dependencies

Downloads reference assemblies from secure storage
Decrypts using ENCRYPTION_KEY
Downloads Thunderstore dependencies from thunderstore.toml
Step 3: Build

Compiles your mod DLL
Runs in isolated environment
Uses exact game assembly versions
Step 4: Package

Creates Thunderstore package structure
Includes DLL, icon, README, manifest
Copies Autoload.Meta.toml if present
Organizes files for mod manager compatibility
Zips everything up
Step 5: Publish

Uploads to Thunderstore using THUNDERSTORE_TOKEN
Creates downloadable release
Updates mod page automatically

Version Management
Use semantic versioning: MAJOR.MINOR.PATCH

MAJOR (1.0.0 → 2.0.0): Breaking changes, incompatible with old versions
MINOR (1.0.0 → 1.1.0): New features, backwards compatible
PATCH (1.0.0 → 1.0.1): Bug fixes, no new features
Example progression:

1.0.0 - Initial release
1.0.1 - Fixed a bug
1.1.0 - Added new feature
1.1.1 - Fixed bug in new feature
2.0.0 - Major rewrite, breaks compatibility
Update version in both thunderstore.toml and your plugin's BepInPlugin attribute!

Verify file is on main branch

Build fails in CI

Compare CI environment to your local setup

Advanced Topics
Mod Dependencies
If your mod requires another mod, declare it in multiple places:

1. In thunderstore.toml:

[package.dependencies]
BepInEx-BepInExPack = "5.4.21"
Robyn-OstraAutoloader = "0.1.10"
bitMuse-LaunchControl = "1.0.0"
Copy
2. In Autoload.Meta.toml (if using JSON data):

[AutoLoad.HardDependencies]
LaunchControl = "1.0.0"  # Must be present

[AutoLoad.SoftDependencies]
OptionalMod = "1.0.0"  # Optional, loads first if present
Copy
3. In your plugin code:

[BepInPlugin("com.yourname.yourmod", "Your Mod", "1.0.0")]
[BepInDependency("com.ostranauts.LaunchControl", BepInDependency.DependencyFlags.HardDependency)]
public class YourModPlugin : BaseUnityPlugin
{
    // ...
}
Copy
Hard vs Soft Dependencies:

Hard: Mod won't work without it - causes error if missing
Soft: Mod has optional features with it - gracefully handles absence
See existing mods like Smarter Hauling for examples.
